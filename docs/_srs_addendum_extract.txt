CHAPTER 21
PDF Generation Specifications
ManufactureERP uses DomPDF (via the laravel-dompdf package) as its primary PDF rendering engine.
Every document type has a dedicated Blade template, a Laravel PDF job class, and a structured delivery
pipeline. This chapter specifies the exact content, layout, data sources, and generation logic for every PDF
in the system.
21.1 PDF Engine Architecture
Technology: barryvdh/laravel-dompdf v2.x  |  Queue: All PDF jobs pushed to pdf-generation queue via Laravel Horizon  | 
Storage: Generated PDFs stored in storage/app/documents/{module}/{year-month}/  |  Delivery: Download link via signed
URL (expires 24h) or direct WhatsApp attachment
1
Controller / Event Triggers PDF
Route or event calls PdfGeneratorService::generate($model). A dedicated job class is dispatched to the queue.
2
Blade Template Rendered
The queued job loads the appropriate Blade view (resources/views/pdfs/{module}/{template}.blade.php) and
passes the Eloquent model with eager-loaded relations.
3
DomPDF Renders HTML → PDF
Dompdf::loadView() converts the HTML+CSS to PDF. Paper size A4, orientation portrait (landscape for reports).
Font: DejaVu Sans (Unicode support for ■ symbol).
4
PDF Stored & URL Generated
PDF saved with hashed filename. Signed URL generated via URL::temporarySignedRoute() expiring in 24 hours.
URL stored on the model record.
5
Delivery
User downloads via browser. OR job fires WhatsApp notification attaching the PDF URL. OR email via Laravel
Mail with PDF as attachment.
Global PDF Header/Footer Blade Component:
All PDFs share a master layout (layouts/pdf-master.blade.php) that injects: company logo (top-left), company
name + GSTIN + address (top-right), document title centred below header, and footer with page number + 'This
is a computer-generated document.' + generation timestamp.

manufacturing company. Each use case names the actors, the trigger, every system step, the documents
generated, and the notifications fired — giving the engineering and QA team a concrete script for
development and testing.

22.4 Use Case: Monthly Payroll Processing
Company
Suncure Pharma Pvt. Ltd.
Pay Period
April 2026
Actors
Kavita (HR Manager), Mehul (Accountant), All Employees
Trigger
Last working day of April — Kavita initiates payroll run
1
Kavita Reviews Attendance
HR → Attendance → April 2026 summary. Reviews daily attendance records. Finds 3 employees with
unapproved absences. Marks 2 as LOP (Loss of Pay), 1 approved as casual leave (pre-approved application).
Attendance locked for payroll.
2
Leave Reconciliation
System reconciles approved leaves against leave balance. Casual leave balance deducted. LOP days calculated
for each employee. HR confirms leave summary: 45 employees, total LOP days = 8 across 3 employees.
3
Kavita Initiates Payroll Run
HR → Payroll → New Run → April 2026. System pre-populates all active employees. payroll_runs record created.
Status: CALCULATING.
4
Auto-Calculation
System computes for each employee: (1) Gross = Basic + HRA + Allowances − LOP deduction. (2) PF Employee
